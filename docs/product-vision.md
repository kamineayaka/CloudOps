# ArchOps 产品愿景与整体思想

> 本文是产品北极星文档：说明 **为什么做、做什么、不做什么**，以及核心难点上的设计立场。  
> 工程实现细节见 `docs/architecture.md`；主线实施计划见 `docs/mainline-implementation-plan.md`；任务清单见 `docs/architecture-refactor-todo.md`、`docs/agent-optimization-todo.md`。  
> 灵感来源：[opskat/opskat](https://github.com/opskat/opskat)（运维工作台），本项目在此之上强调 **可共享、可演进的组织级运维记忆**。

---

## 1. 一句话定位

**ArchOps** 是一个 B/S 架构的 AIOps 控制面：用户登记资产（Linux 服务器、K8s 集群等），通过对话驱动 Agent 在权限与审批约束下执行运维动作；**所有 Agent 共同维护一份持续更新的 Architecture 知识**，并以工作日志与 RAG 检索增强后续决策的准确性。

它不是「带聊天框的 SSH」，而是 **资产控制面 + 可治理的执行闭环 + 活的架构记忆**。

---

## 2. 产品主线（用户价值闭环）

```
登记 / 分组资产
      ↓
与 AI 对话（绑定目标资产）
      ↓
按角色风险：自动执行 或 人工审批后执行
      ↓
从执行与观察中提炼架构事实
      ↓
更新组织级 Architecture + 追加 Agent 工作日志
      ↓
RAG 检索反哺下一次对话与执行
```

### 典型故事

1. 用户添加三台 Linux 服务器，放入分组 `Hadoop`。  
2. 对话：「这是一个 Hadoop 集群，帮我读日志并找出问题。」  
3. Agent 在授权范围内探测与诊断，识别例如：node1=NameNode，node2/node3=DataNode。  
4. 将这些 **架构事实** 写入共享 Architecture（而非只留在当次对话上下文）。  
5. 后续对话涉及 Hive、Spark、Docker 等时，Agent 通过 RAG 命中已沉淀的拓扑与角色，减少重复探测、提高命令针对性。

---

## 3. 三大知识对象（必须分清）

| 对象 | 性质 | 谁写 | 生命周期 |
|------|------|------|----------|
| **Architecture（架构文档）** | 组织级 **单一事实来源（SSOT）**：资产内结构、资产间拓扑、组件角色、关键约定 | 人工编辑 + Agent **提议/经确认后写入** | 持续演进；有版本；可按资产/分组分区 |
| **Work Log（工作日志）** | 某次会话/Agent 的 **过程记录**：执行了什么、观察到什么、结论是什么 | 该会话 Agent 追加写入 | 追加为主，不覆盖 Architecture |
| **Audit Log（审计链）** | **合规与追责**：谁在何时对何资产做了何敏感操作 | 系统强制写入 | 只追加、可校验哈希链 |

原则：

- **日志是过程，Architecture 是共识状态，审计是证据。** 三者不可混用。  
- RAG 主要检索 **Architecture 切片 + 相关 Work Log**；审计默认不进入模型上下文（除非明确的合规查询场景）。

## 4. 核心难点与设计立场

以下难点若不事先定规则，共享 Architecture 会变成「全局幻觉库」。下文给出本项目的立场。

### 4.1 写回不可靠（模型把猜测当事实）

**风险：** Agent 误判角色或拓扑并写入 SSOT，后续所有对话被污染。

**立场：**

1. **默认：提议更新（Proposal），而非静默覆盖。**  
   Agent 产出「架构变更提案」→ 展示 diff → **OPERATOR/ADMIN 确认** 后才合并进 Architecture 并触发再索引。  
2. **允许自动合并的窄通道（可配置）：**  
   - 仅限 **低风险、高可验证** 的事实（例如：从明确命令输出解析出的 hostname、已存在资产的包版本）；  
   - 必须带来源：`command` / `stdout 摘要` / `assetId` / `置信度`；  
   - 高影响断言（角色变更、集群归属、跨资产拓扑）**必须人工确认**。  
3. **每条架构事实带 provenance（出处）。** 无出处的内容不得进入 SSOT。  
4. **支持回滚到上一版本。** 错误写入可一键撤销并 reindex。

### 4.2 一份全局文档膨胀与冲突

**风险：** 多会话同时改「一篇超长 Markdown」，冲突、不可读、检索变差。

**立场：**

1. **逻辑上是一份 Architecture，物理上按分区存储。**  
   推荐分区键：`资产组（如 Hadoop）` / `单资产` / `全局拓扑`。对外仍呈现「统一架构视图」。  
2. **结构化优先，散文为辅。**  
   - 核心字段用结构化记录（角色、组件、依赖边、标签）；  
   - 长叙述作为 section 正文，便于人读与引用。  
3. **乐观并发 + 合并策略：**  
   - 更新基于版本号；冲突时人工解决或按分区合并；  
   - 禁止无版本的全量覆盖写。  
4. **检索粒度是 chunk/分区，不是整本塞进 prompt。**

### 4.3 「什么算架构变更」难以判定

**风险：** 每个 `df`、`tail` 都触发写 Architecture，噪声爆炸；或真正拓扑变化漏写。

**立场：三级事件分类**

| 级别 | 含义 | 示例 | 对 Architecture |
|------|------|------|-----------------|
| **L0 只读观察** | 不改变认知模型 | 看负载、跟一下已知日志路径 | 只写 Work Log |
| **L1 认知发现** | 新发现或修正架构认知 | 确认 NN/DN、发现 Hive 元存储位置 | 生成 Proposal（默认真人确认） |
| **L2 真实变更** | 环境被改动 | 扩容节点、改配置、部署组件 | Work Log + 强制 Proposal + 通常更高审批 |

Agent 工具层与系统提示必须显式区分 L0/L1/L2；**禁止**把 L0 提升为架构写入。

### 4.4 工作日志与架构事实混淆

**立场：**

- Work Log：**时间序、追加、允许含假设与中间结论**（标注为 hypothesis 亦可）。  
- Architecture：**当前有效共识**，只保留被接受的事实与明确标注的不确定项（`unverified`）。  
- 从日志「晋升」为架构事实，必须走 Proposal 流水线，不能隐式拷贝。

### 4.5 权限与可见性

**立场：**

1. **执行权限与知识权限分离考虑：**  
   - 能否 SSH/执行工具：走 RBAC + 审批；  
   - 能否查看/搜索某分区 Architecture：走资产范围（如 `user_assets` / 分组 ACL）；  
   - 能否批准架构提案：至少 OPERATOR，高影响分区可要求 ADMIN。  
2. **Agent 只能更新其有权访问的资产相关分区。** 越权提案直接拒绝。  
3. VIEWER：可读允许范围内的 Architecture / 日志检索；不可执行、不可批准提案。

---

## 5. Agent、权限与执行治理

1. **对话绑定目标资产（或资产组）**，工具默认在此范围内作用。  
2. **风险分级 + 审批门：** 高风险命令先批后跑；与架构 Proposal 审批可复用同一治理理念，但对象不同（「执行」vs「改知识」）。  
3. **所有敏感执行进审计链**；架构合并事件也打审计（谁批准了哪条事实）。  
4. 前端保持薄客户端：风险判断、提案合并、ReAct 循环在服务端。

---

## 6. RAG 在本产品中的位置

RAG 不是装饰，而是规模化后的必需品：

1. **索引范围：** Architecture 分区正文与结构化摘要、已接受事实、相关 Work Log。  
2. **查询时机：** 每轮对话构建上下文时检索 Top-K；工具执行前可对目标资产加一次定向检索。  
3. **更新时机：** Architecture 合并或日志落库后异步再索引；切换 embedding 模型后全量 reindex。  
4. **反模式：** 把整份 Architecture 无差别塞进 system prompt；用审计全文做日常 RAG。

目标效果：Agent **先检索已知拓扑，再决定是否探测**，避免每次对话重复「这是什么集群」。

---

## 7. 与 opskat 类项目的差异

| 维度 | 运维工作台（如 opskat） | ArchOps 目标 |
|------|-------------------------|---------------|
| 重心 | 终端、工具、单人效率 | 控制面 + 组织记忆 |
| 记忆 | 多在会话内 | 跨会话共享 Architecture |
| 治理 | 弱或本地 | RBAC、审批、审计 |
| 知识 | 可选 | Architecture + Work Log + RAG 为一等能力 |

借鉴其「把运维动作做得顺手」，但北极星是 **可治理的 AIOps 记忆闭环**。

---

## 8. 非目标（当前不做）

- 多租户 SaaS 隔离（除非产品战略变更）  
- 用微服务拆分来「表示」模块化（先把单体边界做对）  
- 让模型拥有无审批的生产写权限  
- 将 Architecture 写成不可回滚的黑盒自动覆盖  
- 为协议而协议的外部 MCP 兼容（内置工具注册表优先；外部适配可后置）

---

## 9. 与当前代码库的关系

已有底座（应保留并服务本愿景）：

- 资产与凭证、SSH 执行与连接池  
- AI 对话 / 工具调用 / 审批  
- `ArchitectureSnapshot`、`WorkLog`、pgvector RAG 的雏形  
- JWT + RBAC + 审计链  

尚需升格为产品核心的能力：

- **架构变更提案（Proposal）流水线** 与版本化分区 Architecture  
- L0/L1/L2 事件分类与写回策略  
- 事实出处（provenance）与回滚  
- 按资产范围过滤的知识检索与更新  
- 「执行后 → 提案 → 合并 → 再索引」的自动编排  

工程任务应优先服务上述缺口；纯重构以不破坏本愿景为准。

---

## 10. 演进原则（给后续开发与 Agent）

1. **先可信，再自动：** 自动写 Architecture 的范围只能由窄变宽，不能反过来。  
2. **先分区结构化，再追求一篇完美长文。**  
3. **每条进入 SSOT 的事实必须可追溯到证据。**  
4. **知识更新与命令执行一样，属于被治理行为。**  
5. **文档与代码一致：** 行为变更时同步更新本文与 `docs/architecture.md`。

---

## 11. 成功长什么样

- 新会话询问 Hadoop 问题时，Agent 能引用已确认的 NN/DN 等事实，而不是从头猜。  
- 错误架构写入可被发现、回滚，且审计可追责。  
- 只读排障不会污染 Architecture；真正的认知升级与环境变更会留下清晰提案与日志。  
- 权限不足的用户既不能乱执行，也不能改写其不可见资产的架构知识。
